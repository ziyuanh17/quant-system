# Semantic-Target Alpaca Paper Request Preparation Design

Date: 2026-06-26

Status: In review

## Purpose

This document designs the next software-infrastructure step: a command that
prepares one reviewed Alpaca paper request without contacting Alpaca or
submitting an order.

The system already has:

- local semantic-paper request generation from a strategy signal;
- local semantic-paper execution;
- a one-request Alpaca paper execution command;
- runtime evidence that the Alpaca paper command is visible.

The missing piece is a safe bridge from reviewed strategy/target artifacts to
the `SemanticTargetAlpacaPaperOperatorRequest` consumed by
`quant semantic-target alpaca-paper`.

## Proposed Command

```bash
quant semantic-target prepare-alpaca-paper-request \
  --request-id reviewed-aapl-alpaca-paper-001 \
  --source-request-path data/semantic-target/local-paper-requests/inputs/requests/reviewed-local-paper-request.json \
  --output-root data/semantic-target/alpaca-paper-requests/reviewed-aapl-alpaca-paper-001 \
  --paper-output-root data/semantic-target/alpaca-paper/reviewed-aapl-alpaca-paper-001 \
  --max-order-notional 1000 \
  --allowed-max-quantity 2 \
  --valid-for-seconds 900
```

The command name says "prepare" because it writes a reviewed request bundle
only. It does not run the request.

## Input Model

The first implementation should consume an existing
`ActivatedSemanticPaperOperatorRequest`. That request already binds:

- contributor set path;
- strategy target decision paths;
- strategy evaluation paths;
- risk policy;
- portfolio target ID and revision;
- risk target ID and revision;
- execution lifecycle policy;
- reference price;
- evaluated time;
- evidence references.

The preparer should load those inputs, verify their hashes where available,
then deterministically materialize the portfolio and risk target decisions
needed by the Alpaca paper request.

This keeps the paper request tied to the same strategy intent that was already
reviewed locally. It also avoids asking an operator to manually write
portfolio/risk JSON.

## Generated Artifacts

The command writes under the chosen output root:

```text
inputs/
  contributor-sets/
  strategy-targets/
  strategy-evaluations/
  portfolio-targets/
  risk-targets/
  requests/
```

The contributor, strategy target, and strategy evaluation artifacts may be
copied by content from the source request inputs. The portfolio and risk target
artifacts are generated by the same deterministic aggregation and risk
evaluation code used by semantic-target orchestration.

The final request is a `SemanticTargetAlpacaPaperOperatorRequest` with:

- schema version;
- safe request ID;
- copied/generated artifact paths and hashes;
- risk policy;
- execution policy;
- reference price;
- live-shaped Alpaca paper safety config;
- request-scoped paper output root;
- `evaluated_at`;
- `valid_until`;
- `alpaca_submission_enabled=True`;
- allowed symbol;
- allowed maximum quantity;
- evidence references linking the source local request.

## Safety Configuration

The generated safety config must be explicit:

```text
mode = live
broker_name = alpaca-paper
live_trading_enabled = true
live_trading_confirmation = I_UNDERSTAND_LIVE_TRADING_RISK
max_order_notional = reviewed CLI value
```

The existing `live` mode is the broker-connected safety gate used for Alpaca
paper. It does not authorize real-money trading.

## Validation Rules

The preparer must fail closed if:

- the source request is missing or invalid;
- the source request has no strategy target decision;
- contributor set and strategy decisions disagree on symbol or unit;
- portfolio aggregation blocks;
- risk evaluation rejects;
- the approved risk target is fractional;
- the approved target exceeds `--allowed-max-quantity`;
- `approved_target * reference_price` exceeds `--max-order-notional`;
- `--valid-for-seconds` is not positive;
- the output request already exists with different content;
- request ID or path components are unsafe.

No rounding is allowed. Fractional research targets must be rejected before an
operational Alpaca paper request is written.

## No Broker Behavior

The preparer must not:

- source `.env`;
- read Alpaca credentials;
- construct `AlpacaPaperBrokerClient`;
- call Alpaca;
- inspect paper account state;
- submit paper orders;
- run local paper;
- run dry-run;
- load launchd;
- run a scheduler.

It is a request-building command only.

## Review Output

The command should print:

- generated request path;
- request ID;
- symbol;
- approved target quantity;
- reference price;
- max order notional;
- valid-until timestamp;
- paper output root;
- source request path.

It should also print a plain warning:

```text
Prepared only. No Alpaca API call was made.
```

## Test Plan

Add source tests proving:

- a reviewed local semantic-paper request can produce an Alpaca paper request;
- generated artifact hashes match;
- fractional approved targets are rejected;
- target quantity above the allowed maximum is rejected;
- request notional above the reviewed maximum is rejected;
- rerunning with the same content is idempotent;
- rerunning with conflicting content fails;
- help text says the command prepares only and does not contact Alpaca.

Focused verification:

```text
.venv/bin/python -m pytest \
  tests/test_semantic_target_alpaca_paper_request_cli.py \
  tests/test_semantic_target_alpaca_paper_cli.py \
  tests/test_semantic_target_alpaca_paper_rehearsal.py
```

Static verification:

```text
.venv/bin/ruff check .
.venv/bin/pyright src/quant tests
```

Run the full non-web suite before merging or using the prepared request in a
runtime paper test.

## Next Gate

After implementation, rehearse the preparer in source with a generated local
semantic-paper request. Then copy or fast-forward the runtime clone and run the
preparer there without sourcing credentials or contacting Alpaca.

Only after the prepared request exists and is reviewed should the system run
one real Alpaca paper command.
